Executive Overview

Instant Pay for GigShifts

Mapping the business goal of competitive instant payouts to the minimal technical lift of the Branch API integration.

Meeting Alignment
  • Audience: GigShifts Executive and Engineering Team (CTO, PM, CFO)
  • Objective: Walk through each step of the integration process, from API setup through disbursement, so your team can accurately scope the technical lift and plan the work.
  • Today's Agenda: API authentication and sandbox setup, worker onboarding via embedded portal, webhook events and lifecycle, disbursement flow, funds settlement, and a summary of the total technical lift including next steps.

GigShifts Context

  • App-based gig staffing platform connecting businesses with 1099 contractors.
  • Currently paying workers via ACH/Direct Deposit on a weekly cadence.
  • In-house backend system for housing roster and payout data.
  • Goal: Catch up to platforms like Uber by adding instant payments as a contractor acquisition and retention benefit.

The Business Case for Instant Pay

Advantages for GigShifts

  • No pre-funding required. Branch pays workers first, invoices GigShifts later via ACH.
  • Eliminates ACH processing costs and other payment fees.
  • Branch handles W-9 collection, earnings tracking, and 1099-NEC/K issuance and delivery.
  • Send payments at any time, including nights, weekends, and holidays.

Advantages for Contractors

  • Payments right after each job, not next week.
  • Fee-free banking with the Branch Wallet and Mastercard debit card.
  • Available to 100% of workers, no credit checks required.
  • Live customer support in English and Spanish.

80% of gig workers say they would choose one platform over another if it paid instantly and without fees. (Source: 2023 Gig Payments Report)

The Technical Reality
  • Embedded Onboarding: Drop a single iframe or WebView into the GigShifts app. Branch hosts all KYC, identity verification, and payout setup screens inside it.
  • Minimal Backend Work: Three server-side API calls (create employee, listen for webhooks, disburse funds) and one client embed. No custom onboarding UI to build.
  • Zero PII Liability: Branch collects SSN, DOB, and address directly from the worker. Sensitive data never touches GigShifts servers.
  • Built-in Duplicate Protection: Every disbursement uses a unique external_id. If the same payment is sent twice, Branch safely ignores the duplicate.
Success Criteria
  • Product (PM): Walk away with a clear integration roadmap that can be planned into the GigShifts product backlog, including the worker onboarding experience end to end.
  • Engineering (CTO): Understand the full security model, API technical lift, and confirm the integration can be scoped within a sprint.
  • Finance (CFO): Confirm that Branch pays workers instantly with zero pre-funding, invoices GigShifts on pre-determined schedule, funding takes place via ACH, and handles 1099 tax compliance.

The Connection

Press Test Connection to execute request.
GET/v1/organizations/100458

Headers

apikey: <secret_key>

Response · 200 OK

{
  "id": 100458,
  "org_key": "gigshifts-sandbox",
  "name": "GigShifts Sandbox"
}
What Just Happened
  • A successful response means your API key is working. This key should only live on your servers, never in source code repositories or client-side applications.
  • The response returns your orgId (100458), which is used in all API endpoint paths. A separate org UUID will be provided by Branch during implementation for the portal embed URL.

Developer Accelerators

  • OpenAPI 3.0.1 Spec: Machine-readable API specification available for code generation and client library scaffolding.
  • Postman Collection: Pre-configured collection for immediate endpoint and webhook event testing.
  • MCP Server: AI-assisted development via the Branch MCP server at docs.branchapp.com/mcp.
  • Interactive Sandbox: Full API sandbox environment for testing all integration calls.

Security Best Practices

  • Secure Key Delivery: Branch delivers API keys and webhook decryption keys through Keeper, a zero-knowledge security vault.
  • Use a Key Management System: Store secret keys in an encrypted KMS. Delete any local copies after securing them.
  • Never commit keys to source control or embed them in client-side applications. Inject as encrypted server-side environment variables.
  • Audit API logs: Monitor for suspicious activity. Contact Branch support immediately to rotate compromised keys.

What Just Happened
  • GigShifts drops one iframe (web) or WebView (mobile) into the app. Everything behind it, identity verification, KYC, legal agreements, and payout setup, is built and hosted by Branch.
  • GigShifts doesn't build any identity verification, KYC, or payout setup screens. Branch owns all of that within the embedded portal, saving your team hundreds of hours of frontend and compliance work.
  • After onboarding, workers who chose an existing debit card can manage their account at direct.branchapp.com/login without needing the Branch app.
  • The Branch Wallet is a digital wallet backed by a Mastercard debit card issued by an FDIC-insured sponsor bank. Workers who choose the wallet get fee-free banking, a physical or virtual card, and access to financial wellness tools via the Branch app.
  • Onboarding through the portal typically completes in minutes for straightforward cases. Branch can confirm exact SLAs during implementation planning.

Step 04

Listen for Webhooks

GigShifts Builds

Branch reports onboarding progress to your backend asynchronously. Step through the events as Jane completes setup.

Webhook Setup (coordinated with Branch during implementation)

  • Provide your webhook endpoint URL to your Branch Customer Support rep (e.g., https://api.gigshifts.com/webhooks).
  • Receive the AES-256-CBC decryption key via Keeper for decoding the encrypted data block in each event.
  • Build a server-side POST endpoint at that URL to receive, decrypt, and process incoming events.
  • Optional: Request Branch IP addresses for webhook delivery whitelisting.

Onboarding Progress

  • 1
    ACCOUNT_VERIFIED_AS_WORKER
    Contact info matched and verified via OTP.
    Action: Allow worker to accrue wages.
  • 2
    WALLET_ACTIVATED
    Fraud checks passed, wallet ready.
  • 3
    PAYMENT_PROFILE_ACTIVATED
    Required Gate
    A wallet or debit card is active. This is the green light: no disbursements are possible until this event arrives.
    Action: Mark worker fundable in payroll.

Payload · 


          

        


        

          What Just Happened
          
    
            
  • GigShifts builds one webhook listener that handles all three events. The only one that gates payments is PAYMENT_PROFILE_ACTIVATED. Until that event arrives, the worker cannot be paid.
    • 
            
  • The outer fields (event, event_id, client_type, client_id) arrive in plaintext. Only the data block is encrypted using AES-CBC with a key provided by Branch via Keeper. Your listener can route and deduplicate before decrypting.
    • 
            
  • Branch may retry delivery, so always check the event_id against previously processed events to avoid handling the same webhook twice.
    • 
            
  • If delivery fails, Branch retries automatically: 30 seconds, 60 seconds, 5 minutes, 30 minutes, then every 60 minutes for up to 7 days.

Funds Flow
Review

        
How GigShifts settles with Branch after instant disbursements are paid to workers.


        

          

            
Disbursement Invoices

            

              

                
GigShifts provides a funding account to Branch. After each ACH window, Branch initiates a priority ACH pull from that account to cover all disbursements paid out during the window.

                
    
                  
  • Account and routing numbers for GigShifts' funding account must be provided to Branch before disbursements go live.
    • 
                  
  • Branch pays workers instantly, then invoices GigShifts via ACH pull. No pre-funding required.
    • 
                

              

            

          


          

            
ACH Pull Schedule

            

              

                
    
                  
  • Invoices created before the 10:45 AM CT cutoff pull funds same-day.
    • 
                  
  • Invoices created after cutoff pull funds the next banking day.
    • 
                  
  • Banking days are every day except weekends and bank holidays.
